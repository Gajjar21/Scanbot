Speed — where does time actually go?

Render cost by DPI (A4 page, typical)

Every OCR call starts with a render. The render cost is quadratic — doubling DPI multiplies pixels by 4, which roughly multiplies OCR time by 4. This is the single biggest speed lever in the whole pipeline.

Settings — what each one actually controls

DPI Settings (biggest impact on speed)

OCR_DPI_MAINe.g. 320

The DPI used for Stage 2 (OCR Main). Every digit in the pipeline starts from here. Lowering from 320→200 cuts Stage 2 time by ~60%. Risk: small or faint text may not resolve. Start any DPI reduction tests at 240 first.

OCR_DPI_STRONGe.g. 420

Used in Stage 3, Stage 4 (rotation), Stage 5 (table), Stage 5.5 (upscale), Stage 5.6 (airway). Reducing this from 420→300 saves ~9s per OCR call across all those stages. Risk: harder-to-read documents may miss. Set to 300 and test on your worst scans.

ROTATION_FAST_DPI200 (default)

Used only in Stage 3.2 (the new fast rotated pass). This is already at a reasonable default. If rotated docs still miss here but catch in Stage 4, bump to 240. If Stage 3.2 is too slow, drop to 180.

ROTATION_PROBE_DPI140 (default)

Used only for the probe (Stage 3.1) — just to determine the angle. Does not need to be high. 140 is a good balance. Raising this makes the probe more accurate but slower. 100 is too low for reliable digit scoring. 160 is the practical ceiling.

Probe / Rotation Certainty Settings

ROTATION_PROBE_CERTAIN_MARGIN300 (default)

Score gap between best and second-best angle needed to call it CERTAIN. At CERTAIN, Stage 4 only tries the one best angle — saves 2 full rotation OCR sequences. If you're seeing docs classified as CERTAIN but matched at the wrong angle, lower this. If UNCERTAIN is too common (Stage 4 running all angles), raise this.

ROTATION_PROBE_LIKELY_MARGIN120 (default)

Score gap needed for LIKELY. Stage 4 still only tries the one angle. Below this is UNCERTAIN. Look at your probe score logs: if you regularly see margins of 80–150, adjusting this threshold can push more files to LIKELY and save 2 rotation sequences each.

ROTATION_PROBE_MIN_FLIP_MARGIN80 (default)

Minimum score gap before the probe will declare a non-zero angle as the base angle. If the non-zero angle beats 0° by less than 80 points, the probe falls back to 0°. Raise this to make the probe more conservative about calling rotation (fewer false rotations). Lower to make it more aggressive.

ROTATION_PROBE_DIGIT_CLEAR_MARGIN24 (default)

If 0° wins on digit count by this margin, skip the expensive text OCR in the probe entirely. This is the fast-path inside Stage 3.1. On most upright documents 0° wins by much more than 24 digits — this fast path fires very often and saves ~3s per file.

Tolerance / Matching Settings

TOLERANCE_HIGH_MAX_DISTANCE2 (default)

Max digit difference allowed when matching a HIGH-confidence candidate to the DB. Distance 1 = one digit wrong (e.g. OCR read 3 as 8). Distance 2 = two digits wrong. Raising to 3 would catch more OCR errors but risks false matches. Only HIGH candidates (found near a label keyword) are allowed distance 2.

TOLERANCE_STANDARD_MAX_DISTANCE1 (default)

STANDARD candidates (found by bare number scanning, no label context) only get distance 1. This is intentionally tight — STANDARD numbers include lots of noise like product codes and phone numbers.

MIN_STAGE_HITS_HIGH_TOL11 (default)

For a HIGH candidate at distance 1 (one digit off), it needs to have been seen in at least 1 stage. This is permissive — a single clear OCR pass near an AWB label with one digit error is accepted. Raising to 2 would require two independent OCR passes to agree.

MIN_STAGE_HITS_HIGH_TOL22 (default)

For a HIGH candidate at distance 2 (two digits off), needs to appear in 2 separate stages. This prevents a single noisy OCR pass from triggering a two-digit tolerance match.

MIN_STAGE_HITS_STANDARD_TOL2 (default)

STANDARD candidates need 2 stage hits before tolerance matching. Even at distance 1 a STANDARD number must be confirmed by two independent OCR passes. Critical safety guard — do not lower below 2.

REQUIRE_SINGLE_STANDARD_CANDIDATE_FOR_TOLTrue (default)

If True, tolerance matching on STANDARD candidates only fires when there is exactly one STANDARD candidate in the pool. If there are two STANDARD candidates, even with 3+ stage hits each, no tolerance match is made — needs review instead. This prevents picking the wrong number when the doc is ambiguous.

Scheduling / Timeout Settings

LONG_PASS_TIMEOUT_SECONDS75.0 (default)

Files in long-pass are interrupted at 75s and their state is saved. They resume in third-pass (no timeout). Raising this means fewer files go to third-pass but each long-pass file blocks new arrivals longer. Lowering means more files hit third-pass. 75s is set to allow Stage 4 at CERTAIN (one angle, ~35-50s) to complete without timeout.

ENABLE_INBOX_TWO_PASSTrue (default)

Enables the three-tier scheduling. If False, every file runs the full pipeline immediately in one pass — simple but slow for batches.

MAX_CONTEXT_RESCUE_MS60000 (default, 60s)

Time cap for the airway label rescue (Stage 5.6). Without this, a 3-crop × 3-rotation airway rescue could take 150+ seconds. 60s is the hard ceiling. If you see [AIRWAY-LABEL] Time budget reached in logs, the rescue was running too long — either the crop regions are wrong for your document type or DPI_STRONG is too high.

Debug — matching symptoms to causes

Scenario: rotated docs are slow

Symptom: rotation_ms is 60–120s. Files with [ROUTE] ROTATED take much longer than [ROUTE] UPRIGHT.

Check these in order:

  1. What is the probe certainty? Look for [ROTATION-PROBE] certainty=. If UNCERTAIN, all 3 non-zero angles run in Stage 4. If LIKELY or CERTAIN, only 1 angle runs. UNCERTAIN with a bad scan = 3× the cost.
  2. Is Stage 3.2 matching early? Look for FastRot* in the method name. If yes, the fix worked and you're on the fast path. If you see [ROUTE] ROTATED but no FastRot in the method, Stage 3.2 found nothing — likely a low-quality scan that needs the full DPI.
  3. What is DPI_STRONG? If it's 420, each rotation pass takes ~14s for digit OCR + ~20s for general OCR. 3 angles = ~100s before you even get to table/upscale. Reducing DPI_STRONG to 300 cuts this to ~50s for 3 angles.
  4. Is ROTATION_PROBE_CERTAIN_MARGIN too high? If most rotated docs have clear probe scores (margin 200–400) but you've set CERTAIN at 500, they all fall to LIKELY or UNCERTAIN and run unnecessary extra angles.

Scenario: high needs-review rate

Symptom: more than 2-3 out of 50 files going to needs-review.
  1. Check the candidate dump. If HIGH candidates are present but not in DB — add them. This is by far the most common cause and has nothing to do with the pipeline.
  2. Check for leading-zero pattern. Candidate 099617498819 when real AWB is 399617498819 = OCR misread. This can still match via tolerance (leading-zero rule is relaxed for tolerance pools). If it's not matching, MIN_STAGE_HITS_HIGH_TOL1 may need lowering to 1 for distance-1 HIGH candidates — it's already at 1 by default so check stage hits count.
  3. Are all needs-review docs of a specific type? If it's always the same document format, the crop regions in Stage 5.6 (airway rescue) may not cover where the AWB sits on that layout.
  4. Is EDM token expired? Files that would catch at Stage 6 go straight to needs-review. Check logs for "Token expired".

Scenario: false matches

Symptom: a file gets matched to the wrong AWB and moved to PROCESSED with an incorrect name.
Note: the pipeline's primary safety guard is the DB check — the matched AWB must exist in the Excel list. A "false match" in the AWB pipeline means the OCR found a different real AWB from the DB — not a made-up number.
  1. Check the method in logs. If method=Filename — the filename contained a different AWB number. Check your Outlook VBA macro naming logic.
  2. If method=Tolerance2-* — a near-miss candidate matched via Hamming distance. If this is a wrong match, the real AWB and the matched AWB differ by only 1-2 digits AND both exist in the DB. Reduce TOLERANCE_HIGH_MAX_DISTANCE to 1 to prevent 2-digit tolerance matches.
  3. If method=EDM-Exists-* — EDM confirmed a candidate that was wrong. This means a wrong number was marked HIGH-confidence (found near a label keyword) and EDM said it exists. Very rare. Check the document to understand what text was near the label.

Scenario: ocr_context_ms is very high (50s+)

Symptom: ocr_context_ms near 60000 in TIMING lines. These are your hardest docs.

ocr_context_ms covers ROI (3.5), Upscale (5.5), Table (5), and Airway (5.6) — all lumped together. To find which stage is dominant:

  • If method contains AirwayLabel → Stage 5.6 matched. The time was in airway rescue. Look for [AIRWAY-LABEL] Time budget reached — if present, the rescue hit the 60s cap before matching.
  • If method contains OCR-Rescue-Upscaled → Stage 5.5 matched. Upscale was the decisive stage — the document is a low-resolution scan that needed the 3× boost.
  • If method contains OCR-Table → Stage 5 matched. Table lines were fragmenting the text — removing them helped. Consider lowering DPI_STRONG slightly as the table pass runs at full DPI.
  • If the file went to needs-review AND ocr_context_ms is 58–60s → airway rescue ran the full 60s and found nothing. The AWB is either in an unusual position on the page or not in the DB.
